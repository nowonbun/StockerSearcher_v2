# AGENTS.md

## 1. 범위와 폴더 책임

- `D:/work/StockerSearcher_v2`
  - 역할: Airflow 기반 데이터 수집 실행 구성 루트입니다.
  - 소유권 경계: 명시적으로 요청된 파일만 수정합니다.
- `src/`
  - 역할: JP/KR 데이터 수집, 모델 생성, 예측 및 PostgreSQL 접근 코드입니다.
  - 소유권 경계: `dataset/`, `create_model/`, `predict/`, `models/`, `entity/`, `function/`의 실행 범위를 관리합니다.
- `dags/`
  - 역할: 시장별 수집 및 예측 DAG 정의입니다.
  - 소유권 경계: 각 시장의 수집 성공 후 일봉·주봉 예측 작업만 실행합니다.
- `src/database/init/`
  - 역할: 빈 PostgreSQL 볼륨에서 `stock` DB와 수집 테이블을 초기화합니다.
  - 소유권 경계: Airflow 메타데이터용 `airflow` DB 스키마를 변경하지 않습니다.
- `src/database/migration/`
  - 역할: 참조 전용 변환 도구입니다.
  - 소유권 경계: 사용자가 명시적으로 요청하지 않으면 DAG에서 실행하거나 수정하지 않습니다.

## 2. 프로젝트 규칙

- 상위 전역 지침과 `D:/work/AGENTS.md`의 적용 가능한 규칙을 우선합니다.
- 데이터 수집 코드는 PostgreSQL `stock` DB를 사용하고 Airflow 메타데이터는 `airflow` DB에 유지합니다.
- DB 설정은 `STOCK_DB_*` 환경 변수로 제공하며 비밀정보를 저장소 파일에 기록하지 않습니다.
- JP/KR 시장별 코드·테이블 차이를 유지하며, 명시적인 중복 제거 이점이 없는 공통화는 수행하지 않습니다.
- 코드 변경 후 문법 검사, Compose 구성 검사, 영향 범위 검증 순서로 확인합니다.

## 3. 워크플로

### PostgreSQL 수집 구성 검증

- 정확한 트리거: `PostgreSQL 수집 구성 검증`
- 실행 순서: 대상 경로 확인 → Python 문법 검사 → `docker compose config` 검사 → 스키마·쿼리 일관성 확인 → 결과 보고
- 중지 조건: 대상 경로, DB 대상(`stock` 또는 `airflow`), 필요한 환경 변수가 모호하거나 누락된 경우
- 실패 보고: `finding`, `evidence`, `next action`을 포함합니다.
- 재실행: 대상 파일 또는 환경 변수 구성이 변경된 경우에만 실행합니다.
- 중복 방지: 동일한 파일 해시와 동일한 환경 변수 이름 집합이면 `no-op`을 보고합니다.

### 일반 요청

- 트리거: 위 정확한 트리거 외의 모든 입력
- 실행: 사용자 요청 범위에서 직접 처리합니다.
- 중지 조건: 상위 지침의 전역 중지 조건을 적용합니다.

## 4. 검증과 보고

- Docker를 실제 기동하거나 DB 데이터를 변경하기 전에는 대상 DB, 자격 증명 전달 방식, 실행 명령을 확인합니다.
- PostgreSQL 초기화 SQL은 빈 볼륨에서만 자동 실행됨을 보고합니다.
- 외부 DB·네트워크·Docker 이미지 빌드 결과를 실행하지 않은 경우 `검증되지 않음`으로 표시합니다.
- 변경 보고에는 수정 파일, 실행한 검증, 남은 위험을 포함합니다.
